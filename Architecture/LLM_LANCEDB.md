**LanceDB Integration Proposal for A.I.M.**  
**Status:** Active Epic
**Date:** May 2, 2026

### 1. Executive Summary: Full LanceDB Native Hybrid Search (Option B)
Currently, A.I.M. utilizes a native LanceDB + Tantivy FTS setup. To eradicate "Global RAG Pollution", we built the **RAG 4.2 Entity-Enforced Intersection**.

While this works perfectly in Python, maintaining a "Split Brain" (LanceDB) creates a massive long-term bottleneck. To ensure enterprise-grade scaling, we are executing **Option B: Full LanceDB FTS**. 

We will migrate BOTH the text content and the vector data directly into LanceDB. This allows us to use LanceDB's underlying Rust-based Tantivy engine to execute true hardware-level Hybrid Search, completely replacing legacy SQLite-based retrieval.

### 2. The Embedding Strategy (Zero Re-embedding Required)
**Crucial Note for the Incoming Agent:** We DO NOT need to change how we generate embeddings, nor do we need to re-embed the existing data through the Ollama API.

The existing SQLite databases (`datajack_library.db`, `project_core.db`) already contain the 768-dimensional `nomic-embed-text` vectors stored as binary BLOBs. The migration script simply needs to read the text and the BLOB from SQLite, decode the BLOB into a vector array, and insert it into LanceDB. Future embeddings will still be generated by the local Ollama daemon exactly as they are today.

### 3. Proposed Architecture
- **SQLite:** Retained ONLY for legacy relational compatibility, timestamps, and project metadata.
- **LanceDB:** The exclusive retrieval engine for all A.I.M. searches (both Semantic and Full-Text).

Instead of running two separate queries in Python, LanceDB handles it natively:
```python
# Example of LanceDB's Native Hybrid Search capability
result = table.search(query_string, query_type="hybrid").vector(query_vector).limit(25).to_pandas()
# OR utilizing the where() clause for strict entity enforcement:
# table.search(query_vector).where(f"text LIKE '%{proper_noun}%'").limit(25)
```

### 4. Migration Path
We will ship an `aim migrate-to-lancedb` command that:
1. Reads existing engram text and pre-computed `nomic` vectors from legacy SQLite archives.
2. Inserts them directly into a new LanceDB instance (`archive/memory_lance/`).
3. Keeps the old SQLite archives intact as a relational backup.
