## 1. The Prompt
Both models were given the exact same, highly-complex Python systems engineering prompt:

> *"Write a fully typed Python 3.14 asynchronous worker pool that utilizes `asyncio.TaskGroup` for concurrency, implements graceful cancellation using `asyncio.CancelledError`, and uses the new `sys.monitoring` API to profile the execution time of each task without adding decorator overhead. Do not use legacy `asyncio.gather`."*

## 2. The Execution

### The Control: Vanilla Gemini (Time: 1 Minute)
Vanilla Gemini operated in a "Zero-Shot" vacuum. It read the prompt and instantly began streaming tokens. Within 60 seconds, it produced a complete `worker_pool.py` script. On the surface, the code looked pristine. It used `TaskGroup` correctly and registered callbacks. A "Vibe Coder" would have merged this into production immediately.

### The Exoskeleton: A.I.M. Agent (Time: 10 Minutes)
Because A.I.M. is bound by the **Core Soul Mandate** (which forbids guessing) and the **Reflex** (which forces it to query the database upon hitting a complex task), it refused to generate code immediately.
1. It ran `aim search "sys.monitoring"` to pull the exact C-API documentation from its [Engram DB](Layered-Engram-Architecture).
2. It wrote a test script (`test_mon.py`) to empirically verify the callback signatures.
3. The test threw a `TypeError`. Instead of hallucinating a fix, A.I.M.'s "Reflex" kicked in. It re-read the documentation, adjusted the `*args` signature, and passed the test.
4. It realized that asynchronous concurrency breaks standard profiling, imported `contextvars`, and finalized the script.

## 3. The Technical Audit (Grok Expert Review)
We submitted both generated scripts to an independent, Expert-Mode LLM (Grok) for a blind technical audit. The results were brutal.

### The Concurrency Data-Race (Vanilla's Fatal Flaw)
**Vanilla Gemini** used the shared `code.co_name` object as the dictionary key to track task time:
```python
if code.co_name in self.target_funcs:
    self._frame_starts[code] = time.perf_counter() # <-- SHARED KEY!
```
*Grok's Verdict:* "Result: classic data race. Task A starts → writes tA. Task B starts (same coroutine) → overwrites with tB. Task A yields → pops tB and records garbage duration. This violates the literal requirement. Vanilla would fail any load test with >1 concurrent task using the same coroutine."

### The ContextVar Brilliance (A.I.M.'s Solution)
**A.I.M.** recognized the trap of the asyncio event loop and isolated the state using Python 3.7+ Context Variables:
```python
current_task_id = contextvars.ContextVar("current_task_id", default=None)
...
def _on_start(...):
    task_id = current_task_id.get() # <-- UNIQUE PER TASK
```
*Grok's Verdict:* "`contextvars.ContextVar` is the *only* correct way to carry task-local state across `await` points. Every `asyncio.Task` gets its own context copy automatically. No shared mutable state, no overwrites. Exactly what production monitoring needs. Winner: A.I.M. 10/10."

### API Overhead & Optimization
*Grok's Verdict:* "Vanilla had no `DISABLE` return anywhere visible... meaning you pay the full instrumentation cost on *every* Python opcode. A.I.M. used `return sys.monitoring.DISABLE` on start, which is the canonical low-overhead pattern. A.I.M. used the API exactly as the 3.14 docs intend."
